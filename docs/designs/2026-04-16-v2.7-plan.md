# v2.7 Architecture → GitHub Bridge Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `@kooleklabs/agentic-app` v2.7 with a new `push-architecture` CLI command that parses v2.6's design artefacts and creates a GitHub Milestone, one feature Issue per `### Feature:` in Acceptance Criteria, and an umbrella tracking Issue — idempotent via HTML-comment markers.

**Architecture:** Pure parser plus `gh` CLI subprocess (no LLM, no SDK call). Six new files under `lib/` following v2.6's small-unit pattern: three pure modules (`marker.js`, `architecture-parser.js`, `issue-body.js`), one subprocess wrapper (`github-repo.js`), one `gh` CLI wrapper (`github-issues.js`), one orchestrator (`github-push.js`). Bin route added to `bin/cli.js`. All subprocess calls use `spawnSync` with array arguments (no shell interpolation).

**Tech Stack:** Node ≥ 18, Jest (from v2.5), ESLint (from v2.5), `js-yaml` (from v2.6), `gh` CLI (new runtime prerequisite for this command only).

**Spec:** [`docs/designs/2026-04-16-v2.7-design-spec.md`](./2026-04-16-v2.7-design-spec.md)

**Prerequisite:** v2.6 must be released. `gh` CLI must be installed and authenticated on the dev machine for local smoke tests (not required for `npm test` — all subprocess calls are mocked).

**Branch:** All work on `feat/v2.7-github-bridge` off `main`. Conventional commits. **No Co-Authored-By Claude trailer** (project preference).

Module exports three pure functions:

```js
const NAMESPACE = 'agentic-app';
const VALID_TYPES = new Set(['feature', 'umbrella', 'adr']);

function slugify(input) {
  if (!input) return '';
  return String(input)
    .trim()
    .toLowerCase()
    .replace(/['`]/g, '')
    .replace(/[^a-z0-9]+/g, '-')
    .replace(/^-+|-+$/g, '');
}

function buildMarker(type, slug) {
  if (!VALID_TYPES.has(type)) {
    throw new Error(`Unknown marker type: ${type}. Expected one of: ${[...VALID_TYPES].join(', ')}`);
  }
  return `<!-- ${NAMESPACE}:${type}:${slugify(slug)} -->`;
}

function extractMarkers(body) {
  if (!body) return [];
  const re = new RegExp(`<!--\\s*${NAMESPACE}:([a-z]+):([a-z0-9-]+)\\s*-->`, 'g');
  const out = [];
  let m;
  while ((m = re.exec(body)) !== null) out.push({ type: m[1], slug: m[2] });
  return out;
}

module.exports = { buildMarker, extractMarkers, slugify, NAMESPACE };
```

## Appendix: Why no real-gh E2E in automated tests

Running the command end-to-end in CI would require:
- A `GITHUB_TOKEN` with create-issue scope (not available on forks / PRs from external contributors)
- Real Issue creation on a test repo every run (cleanup / rate-limit risk)
- Flaky network behaviour

Strategy instead: full mock coverage in `github-issues.test.js` + `github-push.test.js`, plus a `--help` smoke in CI. Manual pre-release E2E against a throwaway repo covers the last mile. Same tradeoff pattern as v2.6's real-SDK gate test.

## Appendix: Subprocess safety

All subprocess calls use `spawnSync('gh', [...args], {encoding: 'utf8'})` — arguments as an array, no shell interpolation. This avoids command-injection risk even when feature names / project names come from `architecture.md` (user-controlled). The `runGh` helper in `github-issues.js` centralises this — no other module invokes subprocesses directly.
