# Design: v2.7 ‚Äî Architecture ‚Üí GitHub Bridge
**Date:** 2026-04-16
**Status:** Approved
**Scope:** `@kooleklabs/agentic-app` ‚Äî a new `push-architecture` command that takes the design artefacts produced by v2.6 and pushes them to GitHub as a Milestone, feature Issues, and an umbrella tracking Issue
**Parent spec:** [`docs/ROADMAP.md`](../ROADMAP.md) ‚Äî bridging release between Phase 1 (design gate) and Phase 2 (GitHub-native automation)

---

## What We're Building

v2.6 (the Architecture Design Gate) produces `docs/architecture.md`, ADRs, and `contracts/api-spec.yaml` on the user's local filesystem. Phase 2's `github-sync` command will consume GitHub Issues to plan+execute features. Between those is a gap: the user has to manually create GitHub Issues from the architecture before Phase 2 is useful.

v2.7 closes that gap. A new `push-architecture` CLI command:
1. Parses the design artefacts produced by v2.6
2. Creates a Milestone, one GitHub Issue per feature in `## Acceptance Criteria`, and one umbrella tracking Issue
3. Links everything together so the team has a single entry point to review the design and pick features to implement
4. Is idempotent ‚Äî re-running on an edited architecture adds new feature Issues without touching existing ones

This release is intentionally **narrow**. Phase 2 (v3.0) consumes its output. Shipping v2.7 first validates the GitHub integration layer before the full Phase 2 orchestration loop.

---

## Why a Separate Release

Two options were considered:

- **Bundle into Phase 2 v3.0** ‚Äî one big release doing both "architecture ‚Üí GitHub" and "Issue ‚Üí code". Bigger scope, bigger PR.
- **Ship v2.7 as a bridge, then v3.0** ‚≠ê chosen ‚Äî two small releases, each independently useful.

The bridge is chosen because:
- It's valuable to users who already use v2.6 ‚Äî they can run it against existing projects without waiting for Phase 2
- It produces the exact Issue shape that Phase 2 will consume ‚Äî dogfooding the data contract before the orchestrator depends on it
- Deterministic, non-LLM code ships fast (1-2 days); Phase 2's orchestration loop will benefit from knowing the Issue shape is fixed

## Component Boundaries

All new files under `lib/`. No existing files modified beyond `bin/cli.js` (routing) and `.github/workflows/ci.yml` (smoke).

| File | Responsibility | Depends on |
|---|---|---|
| `lib/github-push.js` | Top-level orchestrator. Parses args, coordinates the flow, prints plan + result banner, handles approval prompt. | Everything below + `gh` CLI |
| `lib/architecture-parser.js` | Parses `docs/architecture.md` into `{features, adrs, openapiPaths}`. Pure function. | `js-yaml` for api-spec, node `fs` |
| `lib/github-repo.js` | Detects `owner/repo` from `git remote get-url origin`. Fails cleanly on non-GitHub origins. Pure. | `child_process` |
| `lib/github-issues.js` | Thin wrappers around `gh api` / `gh issue` / `gh milestone` calls. Returns normalised objects. | `gh` CLI subprocess |
| `lib/marker.js` | Generates + parses HTML-comment markers that identify items created by this tool. Pure. | (none) |
| `lib/issue-body.js` | Builds Markdown bodies for feature Issues and umbrella Issue from parsed inputs. Pure. | (none) |

Same small-unit pattern as v2.6 (`architect-prompt.js` + `validate-outputs.js` + `architect-gate.js`): parsers are pure and unit-testable; only `github-issues.js` touches the outside world.

## Parser (architecture-parser.js)

Deterministic Markdown split. No LLM. Given the v2.6 validator guarantees the `## Acceptance Criteria` header exists with `### Feature: ...` subsections, the parser relies on that shape.

For each feature block:
- `name` = text after `Feature: `
- `slug` = kebab-case of name
- `criteria` = list of lines matching `^- (Given|When|Then) `
- `relatedPaths` = OpenAPI paths mentioned in the block (simple `/api/...` regex match)
- `relatedAdrs` = ADR file paths mentioned in the block

On malformed input (e.g., `## Acceptance Criteria` missing, no `### Feature:` subsections), surfaces a clear error ‚Äî does not fall back silently.

## Idempotency (markers)

Every Issue created by this tool ends with an invisible HTML comment:

```markdown
<!-- agentic-app:feature:low-stock-alert -->
```

Parts:
- `agentic-app` ‚Äî namespace (distinguishes from other tools)
- `feature` ‚Äî type (`feature`, `umbrella`, `adr` reserved for later)
- `low-stock-alert` ‚Äî slug (kebab-case of feature name)

On re-run, `github-issues.js` lists open Issues, greps bodies for `agentic-app:feature:*` markers, builds `Map<slug, issueNumber>`:
- Features in `architecture.md` with slug in the Map ‚Üí **skip**
- Features not in the Map ‚Üí **create**
- Markers in the Map whose slug isn't in current `architecture.md` ‚Üí **warn** (no auto-close)

Never updates existing Issue bodies. User edits (assignees, manual notes) are preserved.

`--force` bypasses the Map and creates everything fresh (user's responsibility to clean up).

## Error Handling

Every error surfaces as a human-readable message with the next action, not a raw stack trace. Pattern matches v2.6's gate failure banner.

| Failure | Behaviour |
|---|---|
| Not a git repo / no `origin` remote | Exit 1: "No GitHub origin detected. Run `git init` and add a GitHub remote first." |
| `origin` isn't github.com | Exit 1: "Origin is not GitHub (found: gitlab.com). This tool targets GitHub. Use GitLab's native tooling instead." |
| `gh` CLI not installed | Exit 1: "Install `gh` (GitHub CLI): https://cli.github.com/" |
| `gh` not authenticated | Exit 1: "Run `gh auth login` first." |
| `docs/architecture.md` missing | Exit 1: "Run `npx @kooleklabs/agentic-app generate` first to produce the architecture." |
| `architecture.md` malformed (no features) | Exit 1: "Architecture has no `### Feature:` entries under `## Acceptance Criteria`." |
| `gh` 403 / insufficient scope | Exit 1: "Need `repo` scope. Run `gh auth refresh -s repo`." |
| Partial failure mid-run | Do NOT roll back. Report which items succeeded / failed. Next run creates the missing ones via idempotency |
| User rejects the plan prompt | Clean exit 0. Nothing created. |

## Testing Strategy

Unit tests per file. Full end-to-end real `gh` calls are NOT in CI (no tokens, no remote state). Manual pre-release smoke against a throwaway test repo.

| File | Test type | What it covers |
|---|---|---|
| `architecture-parser.test.js` | Unit (pure) | Valid ‚Üí expected output; malformed ‚Üí clear errors; edge cases (zero features, slug collisions) |
| `marker.test.js` | Unit (pure) | Generate + parse roundtrip; slug normalisation; detecting markers in Issue bodies |
| `issue-body.test.js` | Unit (pure) | Feature body, umbrella body, correct links, marker at end |
| `github-repo.test.js` | Unit (mocked `child_process`) | Extracting owner/repo from https/ssh URLs; non-GitHub detection |
| `github-issues.test.js` | Unit (mocked `gh` spawn) | List + filter by marker, create milestone/issue, handle `gh` exit codes |
| `github-push.test.js` | Integration (mocked everything) | Dry-run, idempotency on re-run, approval-prompt, partial-failure recovery |

CI extension: new step in the `verify` job asserts `push-architecture --help` documents `--dry-run`, `--force`, `--milestone`, `--no-umbrella`.

Coverage floor stays at 50% lines (v2.5 starting point).

## Acceptance Criteria

- [ ] Running `push-architecture` on a repo with v2.6 outputs creates: 1 Milestone, N feature Issues, 1 umbrella Issue ‚Äî all linked and marker-tagged
- [ ] `--dry-run` prints the plan with zero `gh` writes
- [ ] Re-run on a clean repo skips existing Issues (matched by marker), creates any new ones, warns on orphans
- [ ] Missing `origin` / `gh auth` / `docs/architecture.md` each exit 1 with an actionable message
- [ ] Non-GitHub origin exits 1 with a clear explanation
- [ ] `--no-umbrella` still creates features, skips umbrella
- [ ] `--force` treats all markers as invalid ‚Üí fresh create
- [ ] Manual pre-release smoke: run against a test repo; Issues render correctly on github.com

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Parser too strict ‚Äî architecture.md variations break it | Parser relies on v2.6's enforced section headers (already validated). Feature subsection parsing falls back to best-effort with warnings |
| User's `gh` token lacks required scopes | Detect the exact scope needed from `gh` error; suggest the refresh command |
| Slug collisions (two features with the same name) | Append `-2`, `-3` etc.; log warning |
| Overwriting user edits to Issue bodies on re-run | Never update existing Issues ‚Äî skip-by-marker means edits are safe |
| GitHub API rate limits | Sequential creation, respect 5000/hr limit; typical run is 10-20 calls |
| Orphaned Issues after architecture edits | Warn only, don't auto-close. User decides |
| Release dependency ‚Äî v2.7 needs v2.6 | v2.6 shipped; users on older versions get an error suggesting upgrade |
