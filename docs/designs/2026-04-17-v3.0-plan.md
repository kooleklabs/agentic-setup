### Task 2: `lib/plan-prompt.js` — build the plan-generation prompt

**Files:**
- Create: `lib/plan-prompt.js`
- Create: `lib/__tests__/plan-prompt.test.js`

Mirrors `lib/architect-prompt.js` from v2.6 — pure prompt construction, no I/O.

Exports:
- `buildPlanPrompt({ issue, architectureSection, apiSpecYaml, adrs, repoName })` → `{ systemPrompt, userPrompt }`
  - `issue` — from `fetchFeatureIssue`
  - `architectureSection` — the `### Feature: <name>` block from `architecture.md` that matches the issue's marker slug (passed in as a string)
  - `apiSpecYaml` — full `contracts/api-spec.yaml` contents, or `null` if absent
  - `adrs` — array of `{ number, slug, body }` for linked ADRs
  - `repoName` — e.g. `kooleklabs/booknook`, for self-identifying context

System prompt specifies the output template (every section from `design-spec.md § Plan Document Format`), forbids code blocks outside `## Files to change`, caps response at ~16k tokens, forbids referencing files/patterns not visible in the prompt.

- System prompt mentions all required sections (`## Problem statement`, `## Files to change`, `## Rollback`, etc.).
- User prompt embeds issue title, issue body, architecture section, API spec, each ADR body.
- When `apiSpecYaml` is `null`, prompt explicitly says "No OpenAPI spec in this project" rather than embedding nothing.
- When `adrs` is empty, prompt says "No linked ADRs."

### Task 5: `lib/gh-plan-pr.js` — create branch, commit, push, open draft PR

**Files:**
- Create: `lib/gh-plan-pr.js`
- Create: `lib/__tests__/gh-plan-pr.test.js`

Exports:
- `openPlanPR({ cwd, slug, planFilePath, planTitle, issueNumber, base, draft })` → `{ prNumber, url, branch }`.

Flow (all `spawnSync`, no shell):
1. `git checkout -b plan/<slug>` — fail if branch already exists locally
2. `git add <planFilePath>`
3. `git commit -m "plan: <slug>"`
4. `git push -u origin plan/<slug>` — fail if remote branch already exists
5. `gh pr create --draft --title "plan: <planTitle>" --base <base> --body "Generated plan for Issue #<N>. See [\`docs/plans/<slug>.md\`](...). Closes or references #<N> — review and merge before implementing."` (omit `--draft` when `draft=false`)
6. Parse PR URL from stdout → return `{ prNumber, url, branch: "plan/<slug>" }`

- [ ] **Step 1: ~6 tests** — happy path (verify exact args), local branch exists → throws, remote push fails → throws, PR URL parsing, `--draft=false` omits flag, post-push restore on failure (best-effort `git checkout -` + `git branch -D plan/<slug>` cleanup, tolerant of errors).

- [ ] **Step 2–5:** run failing → implement → green + lint → commit `feat: add gh-plan-pr subprocess wrapper`.

## Chunk 3: Orchestrator + CLI wiring

### Task 6: `lib/github-sync.js` — orchestrator

**Files:**
- Create: `lib/github-sync.js`
- Create: `lib/__tests__/github-sync.test.js`

Exports `runGithubSync(opts)` where `opts = { cwd, issueNumber, dryRun, force, noComment, base, model, draft, autoApprove }`.

Flow:
1. `checkGhAvailable` → `detectGithubRepo` → `fetchFeatureIssue`
2. Validate: Issue's `markerSlug` matches a feature in `parseArchitecture(cwd).features`. If not → `{ok: false, error}` with guidance.
3. Extract matching architecture section as a string (split `architecture.md` on `### Feature:` and grab the right block).
4. Read `contracts/api-spec.yaml` if it exists.
5. Read each linked ADR body.
6. Check `docs/plans/<slug>.md` existence — if present and `!force` → fail with clear guidance.
7. Check `plan/<slug>` branch (local and remote) — if exists and `!force` → fail.
8. Print plan summary; if `dryRun` → print generated prompt and exit with `{ok: true, dryRun: true}` (no SDK call).
9. Approval prompt (unless `autoApprove !== undefined`).
10. Build prompt via `plan-prompt.js`; invoke `claude-runner.js`'s `runClaudeOneShot` (or whatever the existing SDK wrapper is called — confirm during implementation) with `model`, `systemPrompt`, `userPrompt`, max output 16k.
11. `validatePlanBody(response)` — on missing sections, retry once. On second failure → return `{ok: false, error: "plan truncated after retry"}`.
12. `renderPlanFile(...)` → `fs.writeFileSync(docs/plans/<slug>.md, ...)` (mkdir -p).
13. `openPlanPR(...)` via `gh-plan-pr.js`.
14. Unless `noComment`, `commentOnIssue({number: issueNumber, body: "Plan PR: #<prNumber>"})`.
15. Return `{ok: true, plan: {path, prNumber, url, tokenUsage}}`.

- [ ] **Step 1: ~10 tests** (all collaborators mocked via `jest.mock`):
  - dry-run — no SDK, no FS write, no PR
  - fresh create — SDK called 1x, plan file written, PR opened, comment posted
  - `--force` — re-creates even when plan file exists
  - `--no-comment` — skips `commentOnIssue`
  - slug mismatch with architecture → `{ok: false, error: ...}`
  - Issue missing feature marker → `{ok: false, error: ...}`
  - plan already exists without `--force` → `{ok: false, error: ...}`
  - branch already exists without `--force` → `{ok: false, error: ...}`
  - LLM truncation retries once, then fails clean
  - user cancels prompt → `{ok: false, cancelled: true}`

Reuse the `jest.spyOn(console, 'log').mockImplementation(() => {})` pattern from `github-push.test.js`.

- [ ] **Step 2–5:** run failing → implement → green + lint → commit `feat: add github-sync orchestrator for v3.0`.

## Appendix: Why no real-SDK E2E in automated tests

`plan-generator` calls Claude. Automated tests mock `claude-runner.js` — the real SDK E2E is gated to Step 3 of the release checklist (manual, with live credentials). Mirrors v2.6's `architect-gate` posture: unit-test the prompt shape + the validator, E2E the behaviour pre-release.

## Appendix: Subprocess safety

All `gh` and `git` invocations go through `spawnSync('<cmd>', [argsArray], { encoding: 'utf8' })` — no shell, no string interpolation, arguments are always passed as an array. This is enforced by reusing the `runGh` pattern from `lib/github-issues.js` and applying the same shape to any new `git` calls in `gh-plan-pr.js`.
