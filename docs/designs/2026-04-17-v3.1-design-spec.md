# Design: v3.1 — Plan → Implementation Pass

**Date:** 2026-04-17
**Status:** Draft — awaiting review
**Scope:** `@kooleklabs/agentic-app` — adds `--execute` mode to `github-sync` that reads a merged plan and runs one implementation pass on a new branch. Plus ergonomics: non-interactive `--yes` flag and smart default-branch detection.
**Parent spec:** [`2026-04-17-v3.0-design-spec.md`](./2026-04-17-v3.0-design-spec.md) — v3.0 produced the plan document; v3.1 consumes it.

---

## What We're Building

v3.0 closed the Issue → plan gap. v3.1 closes the next one: turn a reviewed-and-merged plan into a first implementation pass, with the human still in the loop for the PR.

A new mode on the existing command:

```bash
npx @kooleklabs/agentic-app github-sync --issue 42 --execute
```

1. Fetches Issue #42, validates the feature marker (reusing v3.0 logic)
2. Reads the merged plan at `docs/plans/<slug>.md` from the current branch
3. Verifies the plan PR is merged (plan branch `plan/<slug>` is gone from origin)
4. Creates a fresh `impl/<slug>` branch
5. Invokes Claude with Write/Edit/Read/Bash tools, passing the plan document + architecture context as the prompt
6. Leaves the resulting changes committed on `impl/<slug>` as a single WIP commit, ready for human review
7. Prints a next-step banner: `git diff impl/<slug>` / `gh pr create --draft` when ready

**The human still opens the PR.** Automating `/self-review` and PR creation is deferred to v3.1.1 / v3.2 — see "What's Next".

### Also shipping (bundled ergonomics)

- **`--yes` / `-y` flag** on both `push-architecture` and `github-sync` — skips the interactive approval prompt. Unblocks CI, cron jobs, scripted flows. The orchestrator already has `autoApprove`; wiring one CLI flag each.
- **Smart `--base` detection** — v3.0 hardcodes `"main"`. Replace with auto-detect via `gh repo view --json defaultBranchRef.name`, overridable via explicit `--base`. Fixes repos with `master` / `develop` / custom defaults.

---

## Why "Implement Only, Human Commits" in v3.1

Three scopes were considered (reprised from the last round):

| Scope | Estimate | Risk | Value |
|---|---|---|---|
| **A. Implement only, human commits** (this spec) | ~1 week | Low — commit + PR are human gates | Proves plan-consumer contract; real code appears on a branch |
| B. Full loop (implement → `/self-review` → auto-PR) | ~2 weeks | Medium — half-finished PRs if agent runs off the rails mid-loop | One-shot Issue → PR flow |
| C. Full loop + retry + budget + rollback | ~3+ weeks | High — many integration points | Production-grade |

**We ship A first.** Same principle as v2.7 / v3.0: each release is independently useful and proves a data contract. A single WIP commit on a branch is easy to inspect, easy to amend, easy to throw away. Once that contract is firm, v3.1.1 layers `/self-review` + PR on top without changing the execution path.

## Command Interface

```bash
# New in v3.1 — implement a merged plan
npx @kooleklabs/agentic-app github-sync --issue 42 --execute

# New in v3.1 — non-interactive for scripts
npx @kooleklabs/agentic-app push-architecture --yes
npx @kooleklabs/agentic-app github-sync --issue 42 --yes
npx @kooleklabs/agentic-app github-sync --issue 42 --execute --yes

# Existing v3.0 — unchanged
npx @kooleklabs/agentic-app github-sync --issue 42            # plan mode
npx @kooleklabs/agentic-app github-sync --issue 42 --dry-run
```

### New flags

- `--execute` — switch mode from **plan** (default) to **implement**. Requires the plan PR to be merged (see "Pre-flight").
- `--yes`, `-y` — skip the `Proceed? [y/N]` prompt. Non-interactive.
- `--impl-branch <name>` — override the default `impl/<slug>` branch name. Rarely needed.
- `--max-cost-usd <amount>` — soft cap for the execute SDK call (default: `5.00`). Aborts before starting if the model's estimated input cost alone would exceed; hard-aborts the stream if accumulated usage exceeds mid-run.

### Flag modifications

- `--base` — default changes from hardcoded `"main"` to auto-detected via `gh repo view`. Explicit value still wins.

### New modules

- **`lib/plan-reader.js`** — `readMergedPlan({ cwd, slug }) → { body, meta }`. Reads `docs/plans/<slug>.md`, strips the header, validates required sections (reuses `validatePlanBody`). Throws with guidance when the file is absent.

- **`lib/impl-generator.js`** — `runImplementation({ planBody, issue, architectureSection, cwd, model, maxCostUsd }) → { ok, filesWritten, commandsRun, tokenUsage } | { cancelled } | { ok: false, error }`. Wraps the SDK call with `tools: preset claude_code` (this agent NEEDS tools, unlike the planner). Captures the list of files written and bash commands run by inspecting the stream. Hard-aborts if accumulated cost exceeds `maxCostUsd`.

- **`lib/gh-impl-branch.js`** — `createImplBranch({ cwd, slug, base }) → { branch }`. Pre-flight checks (branch doesn't exist locally/remotely), `git checkout -b impl/<slug>` from `base`. Error cases surface to orchestrator.

- **`lib/gh-impl-branch.js`** also exports `commitImplChanges({ cwd, branch, issueNumber, planBodyTitle }) → { sha }` — `git add -A && git commit -m "wip(impl): <title> (refs #<N>)"` on the current branch. No push; human controls that.

- **`lib/default-branch.js`** — `detectDefaultBranch({ owner, repo }) → string`. `gh repo view --json defaultBranchRef -q '.defaultBranchRef.name'`. Used by both `github-push` and `github-sync` to replace the hardcoded `"main"`.

- `lib/marker.js` — unchanged

None. Same SDK, same `gh`, same `js-yaml`.

## Implementation Prompt Structure

Unlike the plan-generator (text-only, no tools), the impl-generator needs full tool access and a different prompt shape.

- **System prompt (Claude Code preset)** — gives the agent the standard file-editing / shell-running behaviour. No changes needed; we just stop overriding it.
- **User prompt** — bundles:
  - Repo identifier (`kooleklabs/booknook`)
  - Issue number + title
  - **The merged plan document body** (verbatim — this is the spec the agent implements against)
  - The matching `architecture.md` feature section
  - The full `contracts/api-spec.yaml`
  - Hard instructions:
    - "Implement the plan above. Write all new/modified files using Write / Edit tools. Run existing tests (`npm test`, `pytest`, etc.) to confirm green. Do NOT create commits — the harness commits your work after you return."
    - "If the plan's `## Implementation steps` reference tooling (build, test, lint) that isn't installed, skip those steps and note it in your final response."
    - "Return a summary of files written and commands run. No code blocks in the summary — the diff is the source of truth."

### Cost controls

- Soft cap on input: if the combined prompt is over ~60k tokens, warn and ask for confirmation (most plans + architecture fit under 30k).
- Hard cap on output: `maxThinkingTokens` + `maxTurns` passed to SDK; abort stream if accumulated cost (tracked per-message) exceeds `--max-cost-usd`.
- `--dry-run` prints the prompt + estimated cost without invoking the SDK (mirrors the plan-gen dry-run).

## Pre-flight Checks

Each check produces a clear `{ok: false, error}` with actionable guidance:

| Check | Failure message |
|---|---|
| `docs/plans/<slug>.md` exists | "No plan found at docs/plans/browse-books.md. Run `github-sync --issue 42` first (without `--execute`) to generate a plan, then merge its PR." |
| Plan file has all required sections | "Plan at docs/plans/browse-books.md is missing: ## Rollback, ## Test plan. Edit the file and retry." |
| Plan PR has been merged | "Plan PR on branch `plan/browse-books` still exists on origin. Merge or close it before running `--execute`." |
| `impl/<slug>` branch does not exist locally | "Branch `impl/browse-books` already exists locally. Delete it (`git branch -D impl/browse-books`) or pass `--impl-branch <name>` for a different name." |
| `impl/<slug>` branch does not exist remotely | Same with `git push origin :impl/browse-books` hint. |
| Working tree is clean | "Uncommitted changes on current branch. Commit or stash before running `--execute`." |

---

## Idempotency and Recovery

- `--execute` is **not idempotent by design**. Each run produces a new implementation attempt on a fresh branch. Re-running without first deleting `impl/<slug>` fails fast (see pre-flight).
- Mid-run SDK failure: whatever files the agent wrote remain in the working tree on `impl/<slug>`. The harness does NOT revert them. User inspects and decides: finish manually, stash, or `git reset --hard main`.
- SIGINT (Ctrl-C): same — agent stops, files so far remain on the branch, no commit is made. Clean exit code 130.
- Cost overage: same — agent stops, partial work preserved for inspection.

---

## Error Handling

- All SDK errors surface as `{ok: false, error}` from the orchestrator — consumer renders the message.
- Git / gh subprocess errors surface verbatim (stderr). Cleanup attempts are best-effort.
- Cost cap is enforced in two places: pre-call estimate (aborts before API calls), mid-stream running total (aborts the SDK query).

## Testing Strategy

Target: ~30 new unit tests across 4 new modules + 2 modified ones.

| Module | Tests |
|---|---|
| `lib/plan-reader.js` | file exists happy-path, file missing, body stripped of header, validator catches truncated plan — ~4 tests |
| `lib/default-branch.js` | success returns name, gh error throws, parses `main` / `master` / unusual names — ~3 tests |
| `lib/gh-impl-branch.js` | `createImplBranch` happy + local-exists + remote-exists, `commitImplChanges` happy + nothing-to-commit, best-effort cleanup on failure — ~6 tests |
| `lib/impl-generator.js` | happy path (fake SDK stream yields assistant text + captures file writes), SDK error, abort, cost-cap abort, prompt includes plan body — ~5 tests |
| `lib/github-sync.js` (execute branch) | routes to execute when `--execute` set, pre-flight fails produce clean errors, integration with commit step, `--yes` bypasses prompt in both modes — ~8 tests |
| `lib/github-sync-cli.js` (parseArgs) | `--execute`, `--yes`, `-y`, `--impl-branch`, `--max-cost-usd` — ~5 tests |
| `lib/github-push-cli.js` (parseArgs) | `--yes` / `-y` — ~2 tests |

Real-SDK E2E (pre-release, manual):
1. Seed a throwaway repo with `push-architecture`
2. `github-sync --issue N` → draft PR → merge
3. `github-sync --issue N --execute` → verify `impl/<slug>` branch with a WIP commit, inspect diff, ensure generated code references the plan's `## Files to change`

## Acceptance Criteria

v3.1 ships when:

1. `github-sync --issue N --execute` against a merged plan produces a local `impl/<slug>` branch with a single WIP commit
2. All 6 pre-flight checks produce clean `{ok: false, error}` with actionable guidance
3. `--yes` / `-y` on both commands bypasses the approval prompt; tested in both plan + execute modes
4. `--base` auto-detects the default branch; `--base <name>` override still works
5. Hard cost cap aborts the SDK stream when exceeded; no partial commits on cost overage
6. Full test suite (existing + new) passes; CI green on node 18/20/22
7. Real-E2E: throwaway repo → push-architecture → github-sync → merge → github-sync --execute → branch with WIP commit matching the plan

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Agent writes unrelated files (scope creep) | Plan document explicitly lists `## Files to change`; prompt says to stay within scope. Post-run, human reviews the diff. |
| Agent fails to run project tests (missing toolchain) | Prompt instructs "skip test steps if tooling isn't present, note in response". Human verifies test status post-commit. |
| Cost blows past `--max-cost-usd` because running total is approximate | Cap is a soft abort — stop as soon as accumulated tokens × price exceeds threshold. Small overage (one extra message) is acceptable; the alternative (precise per-token metering) is noisy. |
| Human forgets to commit their own approvals and just pushes the WIP commit | WIP commit message starts with `wip(impl):` so it's obvious it needs cleanup. Banner tells them to amend or squash before PR. |
| Agent invokes destructive commands (`rm -rf`, `git push --force`) | Tool-use behaviour inherits from Claude Code preset — same safety posture as existing `generate` / architect-gate. `env: { ENABLE_SECURITY_REMINDER: '0' }` is consistent with those. |
| Double-booking: user runs `--execute` while plan PR still open | Pre-flight check #3 catches this; hard fail with guidance to merge or close first. |

## Open Questions

1. **Should `--execute` auto-run test suites after file writes?** Answered: yes, via the agent's Bash tool, driven by the plan's `## Test plan` section. No separate harness step.
2. **Should the WIP commit message include the Issue body summary?** No — keep it short and factual. The plan body is the source of truth and it's already on disk.
3. **Should we support `--execute --dry-run`?** Yes — prints the prompt + cost estimate without running the SDK. Already captured in flag table.
4. **Can `--execute` and `--dry-run` combine meaningfully?** Yes — same as plan mode: show the prompt + estimate, don't invoke.
5. **Should the impl PR body auto-link to Issue #N with `Closes #N`?** Deferred to v3.1.1 which adds auto-PR. For v3.1, the human writes the PR body.

## Out of Scope for v3.1 (deferred)

- **`/self-review` integration** — runs in v3.1.1. For now, humans invoke their own review.
- **Auto-PR creation** — also v3.1.1. Current output stops at local branch with WIP commit.
- **Retry on test failure** — v3.1.2 or later. Current behaviour is one-shot; the agent gets one run and writes what it writes.
- **Project-board automation** — v3.2, unchanged.
- **Multi-Issue / epic execution** — v3.3+.
- **Rollback automation** — v3.3+. Humans revert branches manually for now.
