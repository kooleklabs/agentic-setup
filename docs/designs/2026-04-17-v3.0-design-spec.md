# Design: v3.0 — GitHub Issue → Implementation Plan

**Date:** 2026-04-17
**Status:** Draft — awaiting review
**Scope:** `@kooleklabs/agentic-app` — a new `github-sync` command that reads a feature Issue created by v2.7's `push-architecture` and produces a reviewed implementation plan as a draft PR
**Parent spec:** [`docs/ROADMAP.md`](../ROADMAP.md) — first Phase 2 release (GitHub-native automation)

---

## What We're Building

v2.7's `push-architecture` gave us the entry point: every feature in `architecture.md` now lives as a GitHub Issue with an `agentic-app:feature:SLUG` marker, linked to an umbrella and a Milestone. v3.0 takes the next step: turn a single feature Issue into a concrete implementation plan the team can review before anyone writes code.

A new `github-sync --issue <N>` command:

1. Fetches Issue #N via `gh api`, validates it has a `feature` marker
2. Extracts acceptance criteria, related API paths, related ADRs, and the `architecture.md` section that produced it
3. Invokes Claude (Sonnet) with a focused plan-generation prompt
4. Writes the plan to `docs/plans/<feature-slug>.md`
5. Opens a **draft PR** titled `plan: <feature-name>` with just that file
6. Posts a comment on Issue #N linking to the plan PR

**The output is a plan document, not code.** The human reviews it in the PR. Once merged, existing tooling (`/feature-dev`, plan-feature skill) takes over for implementation. Agent-driven execution of the plan is explicitly **out of scope for v3.0** — see "What's Next" for the v3.1 roadmap.

---

## Why "Planner Only" in v3.0

Three scopes were considered for v3.0:

| Scope | Estimate | Risk | Value delivered |
|---|---|---|---|
| **A. Planner only** (this spec) | 3–5 days | Low — LLM output reviewed by human before anything else runs | Closes the Issue → plan gap; proves the round-trip from v2.7 markers |
| B. Planner + agent execution | 2–3 weeks | Medium — half-finished PRs if agent runs off the rails | Adds code but still needs manual review |
| C. Full Phase 2 loop (plan + execute + self-review + board sync) | 4–6 weeks | High — many integration points, long feedback cycle | One-shot Issue → merged PR, but opaque when it goes wrong |

**We ship A first** for the same reason v2.7 existed as a bridge: each increment is independently useful and proves a data contract (this time: the plan document shape) that later releases depend on. v3.1 layers agent execution on top once the plan shape is firm. v3.2 adds board sync.

## Command Interface

```bash
# Primary flow
npx @kooleklabs/agentic-app github-sync --issue 42

# Flags
--issue <N>              # REQUIRED — Issue number to sync
--dry-run                # print extracted context + generated prompt, no LLM call, no PR
--force                  # re-generate even if docs/plans/<slug>.md already exists
--model <name>           # override (default: claude-sonnet-4-6)
--base <branch>          # base branch for the plan PR (default: main)
--no-comment             # skip the "plan PR: #M" comment on Issue #N
--help
```

Wired into `bin/cli.js` alongside `init`, `generate`, `migrate`, `push-architecture`. Routes to an async handler in `lib/github-sync-cli.js` (following the `push-architecture` pattern — thin CLI wrapper keeps `lib/github-sync.js` process-free for testing).

### New modules

- **`lib/github-issue-reader.js`** — `fetchFeatureIssue({owner, repo, number}) → {number, title, body, markers, acceptanceCriteria, apiPaths, adrRefs}`. Thin wrapper around `gh api`, validates feature marker presence, parses the body the same way `lib/architecture-parser.js` does. Throws if Issue is missing the marker or is a PR.

- **`lib/plan-generator.js`** — `generatePlan({issue, architectureSection, apiSpec, adrs, model}) → { body, tokenUsage }`. Builds the LLM prompt from the extracted context, calls the Agent SDK, returns the plan body as a Markdown string. Retries once on truncation.

- **`lib/gh-plan-pr.js`** — `openPlanPR({owner, repo, slug, planBody, issueNumber, base}) → {prNumber, url}`. Does: create branch `plan/<slug>` locally, commit the plan file, push, `gh pr create --draft`, return PR info. Separate module so it's easy to mock in orchestrator tests.

- **`lib/github-sync.js`** — orchestrator. Same shape as `lib/github-push.js`: try/catch wrapping, `{ok, error}` result, dry-run branch, printPlan/printBanner helpers.

- **`lib/github-sync-cli.js`** — CLI wrapper with `parseArgs` + `main(argv) → exit code`. Mirrors `lib/github-push-cli.js`.

### Reused from v2.6/v2.7

- `lib/github-repo.js` — detect `{owner, repo}` from git remote (unchanged)
- `lib/marker.js` — `extractMarkers(body)` to validate the feature marker (unchanged)
- `lib/architecture-parser.js` — `parseArchitecture(cwd)` to get the architecture section, API spec, ADR list (unchanged)
- `lib/claude-runner.js` — Agent SDK invocation (unchanged; same pattern as v2.6's architect-gate)
- `lib/github-issues.js` — `checkGhAvailable`, `updateIssue` for the Issue comment (add one wrapper: `commentOnIssue`)

### New runtime dependencies

None. Reuses `js-yaml` and `@anthropic-ai/claude-agent-sdk` already declared.

## Plan Document Format

`docs/plans/<slug>.md` — consistent shape so downstream `/feature-dev` can consume it mechanically:

```markdown
# Plan: <Feature Name>

**Source:** [Issue #42](../../../issues/42) · `<repo-url>`
**Generated:** 2026-04-17 by `@kooleklabs/agentic-app@3.0.0`
**Model:** `claude-sonnet-4-6`

## Problem statement

<2–3 paragraphs paraphrasing Issue body + acceptance criteria>

## Acceptance criteria

- Given ...
- When ...
- Then ...

## Approach

<Claude's recommended design, referencing the architecture + ADRs>

## Files to change

- `path/a.ts` — <what changes and why>
- `path/b.ts` — new file for ...

## Implementation steps

1. <Step 1 — small, testable>
2. <Step 2 — ...>

## Test plan

- Unit: <specific cases>
- Integration: <specific scenarios>
- Manual: <what to click through>

## Open questions

- <Anything Claude was unsure about>

## Rollback

<How to undo this change if it goes wrong>
```

The template is strict because `/feature-dev` will consume the `## Implementation steps` and `## Files to change` sections programmatically in v3.1. Deviations from the shape will get caught by a simple validator (`lib/validate-plan.js` — future work, but mentioned so we don't paint into a corner).

## LLM Integration

- **Model default:** `claude-sonnet-4-6` (same default as `generate`). Override via `--model`.
- **Prompt structure:** a single-turn user prompt embedding (a) the feature Issue body, (b) the matching `architecture.md` section, (c) the full `api-spec.yaml`, (d) the full bodies of linked ADRs. System prompt specifies the output template and forbids code blocks outside the "Files to change" section.
- **Token budget:** expected ~8k input, ~4k output for a typical feature. Hard cap at 16k output to keep costs bounded.
- **No tools:** the generator doesn't use file-reading tools — everything it needs is in the prompt. This keeps runs deterministic and avoids filesystem accidents.
- **Retry:** one retry on truncation (detected by absence of the `## Rollback` closing section). Second failure → fail with guidance.

---

## Idempotency Strategy

- If `docs/plans/<slug>.md` already exists and `--force` is not passed, fail early with a clear message: "Plan already exists — pass `--force` to re-generate, or delete the file first."
- If `plan/<slug>` branch already exists remotely, fail early with "plan branch already pushed — delete it with `git push origin :plan/<slug>` or pass a different Issue number."
- No HTML-comment marker needed in the plan file — the filename *is* the marker (one plan per feature slug).

---

## Error Handling

Every external call is wrapped; the orchestrator returns `{ok: false, error}` rather than throwing. Callers (CLI wrapper, tests) render cleanly.

- **Issue not found** → "Issue #42 not found in kooleklabs/booknook."
- **No feature marker** → "Issue #42 doesn't have an `agentic-app:feature:*` marker. Was it created by `push-architecture`?"
- **Marker slug doesn't match any feature in `architecture.md`** → "Issue #42's slug `browse-books` is not in docs/architecture.md. Did you edit the architecture without re-running push-architecture?"
- **Plan already exists** → see idempotency above.
- **LLM call fails** → surface SDK error, no partial writes.
- **Branch/PR creation fails** → plan file is NOT committed locally; user sees the file in the working tree and can retry.

## Testing Strategy

Target: ~25 new unit tests across 5 new modules, all with mocked `gh` + mocked Agent SDK.

| Module | Tests |
|---|---|
| `lib/github-issue-reader.js` | fetch + parse success, missing marker, PR-skip, invalid slug, gh error — ~6 tests |
| `lib/plan-generator.js` | prompt shape assertions, truncation retry, token-cap enforcement, model override — ~5 tests |
| `lib/gh-plan-pr.js` | branch creation, push, `gh pr create --draft` args, PR number parsing, gh error — ~5 tests |
| `lib/github-sync.js` (orchestrator) | dry-run, fresh create, `--force`, missing plan-dir creation, idempotency, Issue-comment posting, `--no-comment`, error paths (reader/generator/PR creation) — ~8 tests |
| `lib/github-sync-cli.js` | `parseArgs` per flag (6 flags + defaults) — ~7 tests |

Real E2E is **user-driven**, same as v2.7: plan author runs against a throwaway repo that has a seeded feature Issue. CI smoke asserts `github-sync --help` documents all flags (follow v2.7 pattern).

## Acceptance Criteria

v3.0 ships when:

1. `npx @kooleklabs/agentic-app@3.0.0 github-sync --issue <N>` produces a draft PR with a plan document conforming to the template above
2. Idempotency: re-running against the same Issue without `--force` fails fast without touching the repo
3. `--dry-run` prints the extracted context and the LLM prompt without making any API call or file change
4. All 5 error paths above produce clean `{ok: false, error: "..."}` results that `bin/cli.js` surfaces with a non-zero exit code
5. Full test suite (existing + new) passes; lint clean; CI green on node 18/20/22
6. Real-E2E against a throwaway repo with a seeded Issue produces a reviewable plan PR

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| LLM produces a plan that doesn't follow the template | Strict system prompt + optional validator in v3.1; worst case, human edits the PR before merging |
| Generated plan references files/patterns that don't exist | Plan generator sees `architecture.md` only; doesn't invent filesystem claims. Human review catches outliers. |
| Token cost creep if feature bodies grow | 16k output cap; telemetry logged to stderr so users see token usage per run |
| `gh pr create --draft` fails on repos where draft PRs are disabled (orgs) | Fall back to `--draft=false` with a warning; document the requirement in README |
| Git commits on the user's behalf feel invasive | The CLI only commits the single plan file to a new branch; never touches main or existing branches. Clear CLI output shows what's about to happen before the push. |

## Open Questions

1. **Should the plan PR be a draft or ready-for-review?** Draft default, flag to override (`--ready`) — answered.
2. **Model default — Sonnet or Opus?** Sonnet, matching `generate`. Opus available via `--model`.
3. **Where do multi-feature plans (umbrella-level design changes) live?** Out of scope for v3.0 — plans are per-feature. Umbrella-level "epic plans" deferred to v3.3+ if needed.
4. **Should the command auto-run `/self-review` on the plan before opening the PR?** No for v3.0 — it's a plan document, not code. `/self-review` is code-focused. Human review suffices.
5. **Does it need to work when there's no `docs/architecture.md` (e.g. the user hand-created an Issue)?** No — v3.0 requires the Issue came from `push-architecture`. Graceful error message covers the mismatch.

---

## Out of Scope for v3.0 (deferred)

- Agent-driven implementation of the plan (v3.1)
- Automated `/self-review` + commit after implementation (v3.1)
- Project board automation (move Issue to "In progress" / "In review") (v3.2)
- Umbrella-level or multi-feature planning (v3.3+)
- Support for Issues not created by `push-architecture` (v3.3+ if demand exists)

## What's Next

After v3.0 ships:

- **v3.1** — add `--execute` flag. After plan PR is merged, `github-sync --issue N --execute` reads the merged plan, runs the implementation via subagent, opens a code PR back-linked to the Issue.
- **v3.2** — project board integration. Issues move across columns as PRs open/merge. Opt-in via `--board` flag.
- **v3.3** — support for ad-hoc Issues (no marker), umbrella-level epic plans.

Phase 2 completes when v3.2 ships. Phase 3 (Master Orchestrator Engine) picks up from there.
